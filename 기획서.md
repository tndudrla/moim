# 🍽 상황 맞춤 식당 추천 AI — 팀 프로젝트 기획서 (v2)

> **하루 완성 / 6인 팀 스프린트용**
> "오늘 어떤 자리인가"를 고르면, AI가 상황에 맞는 식당을 **이유와 함께** 추천하고 지도에 보여주는 전사 공유 웹앱.

---

## 0. 핵심 컨셉 — 두 개의 모드

이 서비스는 성격이 다른 두 상황을 **모드로 분리**한다. 엔진(네이버 검색 + Claude)은 공유하고, 프리셋·우선순위·화면 톤만 다르다.

```
          오늘 어떤 자리인가요?

   🍷 접대용 (손님 모시기)      🎉 회식용 (우리끼리)
   해외파트너·사장님·임원        부서회식·뒤풀이·워크샵
```

| | 🍷 접대용 | 🎉 회식용 |
|---|---|---|
| 상황 | 손님을 모신다 (접대·의전) | 우리끼리 먹는다 |
| 진입 | **프로필 카드 선택** | 상황 + 예산 슬라이더 |
| 우선순위 | 격식·룸·검증된 곳·안전 | 가성비·단체석·다양성·재미 |
| 예산 | 높음 (가격보다 "격") | 1인 금액 딱 맞춤 |
| 특수 기능 | 전사 검증 후기 표시 | 투표·룰렛 |
| 분위기(색) | 차분한 골드/네이비 | 밝고 경쾌 |

> 같은 후보라도 Claude 프롬프트의 모드에 따라 다르게 추천 ("접대: 룸 있고 조용" vs "회식: 가성비 좋고 왁자지껄").

---

## 1. 한 줄 소개

상황(접대/회식)과 대상을 고르면, **AI가 회사 상황에 맞는 식당을 이유까지 붙여 추천**하고 지도에 보여주며, 전사가 검증 노하우를 공유하는 사내 플랫폼.

- **문제**: 접대·회식 장소 정하기 = 조건 많고(대상·예산·격식·거리) 매번 처음부터 고민. 접대 노하우가 개인에게만 쌓임.
- **해결**: 상황별 프로필로 조건 자동 세팅 + AI가 이유까지 설명 + 전사가 검증 결과 공유.

---

## 2. 🍷 접대용 모드 — 프로필(모시는 대상)

프로필 = 미리 세팅된 조건 묶음. 사용자는 대상만 고르면 나머지(예산·룸·음식·특이사항)가 자동 세팅됨.

| 프로필 | 예산 | 격식/룸 | 음식 성향 | 특이사항 |
|---|---|---|---|---|
| 🌏 **해외 파트너** | 높음 | 룸·입식 필수 | 한식(한국 경험) | 외국어 메뉴, 할랄/베지테리언, 젓가락 부담 고려 |
| 👔 **사장님/대표** | 최상 | 프라이빗 룸 | 고급 한정식·오마카세 | 발렛·주차, 조용함, 검증된 곳만 |
| 🏢 **본부 임원** | 상 | 룸 선호 | 격식 있는 한/양식 | 접근성, 적당한 격식 |
| 🤝 **거래처 실무진** | 중상 | 조용한 홀 | 무난한 한식·고깃집 | 대화 가능한 소음 |

**UX**: `[🌏 해외 파트너]` 선택 → 출발지·인원만 추가 → 끝. Claude가 프로필 맥락까지 이유에 반영.

## 3. 🎉 회식용 모드

| 항목 | 값 |
|---|---|
| 상황 | 부서 회식 / 팀 뒤풀이 / 워크샵 |
| 예산 | 1인 금액 슬라이더 (딱 맞춤) |
| 음식 종류 | 한/중/일/양 켜고 끄기 |
| 인원 | 단체석 판단 |
| 도보 시간 | 회사에서 5/10/15분 이내 |
| 특수 | 후보 중 팀원 **투표**, **랜덤 뽑기 룰렛** |

## 4. 전사 공유 = 접대 노하우 DB

단순 추천을 넘어 회사의 집단 지식이 축적됨.

```
[A본부장이 해외파트너 모시고 ○○한정식 다녀옴]
  → "룸 좋았음, 외국인 반응 굿, 1인 5만원" 기록
       ↓
[다른 부서가 같은 상황]
  → "이 집 A본부 해외파트너 접대 후 만족" 표시
```

- 전사가 검증한 식당 목록 / 프로필별 추천 / 사용 기록·후기
- **이 저장소의 Supabase 재활용** → 공유 DB 인프라를 새로 안 짜도 됨.

## 6. 처리 흐름 (공통 엔진)

```
[홈: 접대/회식 모드 선택]
  → [프로필/조건 세팅]
  → [네이버 지역검색 API]   음식분류별 검색으로 후보 수집
  → [도보시간 필터]         회사 좌표 ↔ 식당 좌표 직선거리 ÷ 80
  → [Supabase]             전사 검증 식당·후기 합류
  → [Claude API]           후보 + 모드/프로필 맥락 종합 → 추천 + 이유
  → [지도 + 카드 + 딥링크 + 후기 남기기]
```

### 결과 카드
```
🌏 해외파트너 추천   🇰🇷 한식   ○○한정식
📍 회사에서 도보 7분 · 1인 5만원대(예상)
"외국 손님이 한국 음식을 경험하기 좋고, 영어 메뉴·프라이빗 룸 있음"
✅ A본부 검증 (2회) · [네이버 지도] [전화]
```

| 부분 | 가능성 | 메모 |
|---|---|---|
| 입력폼 + 결과카드 UI | 🟢 95% | 일반 웹 프런트 |
| 두 모드 분리 + 프로필 프리셋 | 🟢 90% | 그냥 설정값. 쉬움 |

| 네이버 검색 후보 수집 | 🟡 70% | 키 발급·인증·파싱 시간 소요 |
| 지도 표시 | 🟡 65% | 네이버 지도 API 연동 |
| 도보시간 계산 | 🟡 60% | 식당 좌표(TM128) 변환이 관건 |

> **눈높이**: "완벽한 서비스"가 아니라 **"모드 고르면 AI가 맥락 맞춰 골라주고 지도에 뜨는 데모가 끝까지 돈다"** 가 목표.

## 8. 핵심 설계 결정

1. **모드 분리(접대/회식)** — 같은 엔진, 다른 프리셋·우선순위·톤.
2. **접대용은 프로필로 조건 자동 세팅** — 사용자는 "누구를 모시나"만 고름.
3. **예약은 딥링크로** — 캐치테이블·테이블링 공개 API 없음 → 각 앱 검색 페이지 연결.
4. **AI는 후보 목록 안에서만 추천** — 환각 방지, 프롬프트로 강제.
5. **예산은 "예상가"로 정직하게** — 카테고리 기반 추정 + 네이버 링크로 확인.
6. **출발지는 회사(고정 주소)** — GPS 불필요, 도보시간 계산 단순화.
7. **전사 공유 DB는 Supabase 재활용** — 인프라 신규 구축 없음.

## 9. 기술 트릭

### (1) 네이버 5개 제한 → 분류별 검색으로 극복
```
켜진 분류 [한식,중식,양식] → "○○ 한식"(5)+"○○ 중식"(5)+"○○ 양식"(5) = 후보 15개
```

### (2) 도보시간 = 직선거리 ÷ 80
```
도보(분) ≈ 직선거리(m) ÷ 80   (한국 부동산 기준: 도보 1분 = 80m)
회사 좌표는 하드코딩(1회 조회), 식당 좌표와의 직선거리로 근사. 길찾기 API 불필요.
```

## 14. 발표 포인트 (스토리)

1. **문제**: 접대·회식 장소 정하기의 피로 + 노하우가 개인에게만 쌓임
2. **해결**: 상황(접대/회식)·대상(프로필) 선택 → AI가 이유까지 붙여 추천
3. **차별점**:
   - 단순 검색 아님 → **"상황 맥락 종합 판단 + 왜 이 집인지 설명"**
   - **전사 접대 노하우를 자산화** (검증·공유)
   - 지도 + 프로필 필터로 시각화
4. **한계 인정**: 예산은 예상가(데이터 한계) → 문제 이해도 어필
5. **데모**: 접대/회식 선택 → 프로필 → 추천 카드 → 지도 → 예약 링크

## 16. v3 확장 — 컴플라이언스 · 출장/해외 · 예약 자동화

### 16-1. ⚖️ 김영란법(청탁금지법) 컴플라이언스 — 접대 모드
- 접대 시 "손님이 청탁금지법 적용 대상(공직자·언론인·사립교직원 등)"이면 **1인 식사 5만원 한도** 적용.
- 체크 시 **한도 초과 식당 자동 제외** + "N곳 제외됨" 안내 + "한도 내" 뱃지.
- 한도 금액은 **현행 5만원**(2024.8.27 3만원→5만원 상향). 코드 상수(`GRAFT_LIMIT`) 한 곳으로 관리 → 개정 대응 용이.
- 확장: 선물 5만원 / 농수산물 15만원(명절 30만원) / 경조사비 5만원 한도도 추가 가능.
- ⚠️ 데모 가격은 "예상가" → 실서비스는 영수증·메뉴가로 정밀 판정 필요.

### 16-2. 🧳 출장 모드 — 지역 선택 (국내 + 해외)
- 출발지를 **본사뿐 아니라 출장지**로 선택 (국내: 강남·부산 / 해외: 도쿄·상하이·자카르타·두바이·뉴욕 …).
- 지역별로 후보·지도 중심이 바뀜. 도보시간·검증DB·프로필 로직은 그대로 재사용.

### 16-3. 🌐 데이터 소스 하이브리드 — 국내 네이버 / 해외 구글맵
| | 국내 | 해외 |
|---|---|---|
| 검색 소스 | 네이버 지역검색 API | **Google Places API (New)** |
| 반경 검색 | 좌표 변환 후 직접 계산 | **좌표+반경 기본 지원** |
| 가격 데이터 | ❌ 없음(예상가) | ✅ **priceLevel/priceRange 제공** |
| 평점·리뷰 | ❌ 없음 | ✅ 제공 |
| 음식종류 | category 문자열 | includedTypes(`japanese_restaurant` 등) |
| 비용 | 무료(일 25,000) | 유료 ≈ $32~40/1,000건, 카드 등록 필요 |

> 핵심: **해외에서는 구글맵이 오히려 가격·평점을 줘서 국내의 "예산 데이터 부재" 약점이 해소됨.**
> 출처: [Google Places API (New) — Nearby Search](https://developers.google.com/maps/documentation/places/web-service/nearby-search)

### 16-4. 🕌 해외 거래처 — 국가별 식문화 고려사항
- 해외 출장지 선택 시 **국가별 식문화 주의사항 패널** 노출 + 추천 이유에 반영.
- 예) 일본(개인룸·건배 매너), 중국(원형테이블·간베이), **인도네시아·UAE(할랄 필수 → 할랄 식당만 필터, 무알콜 안내)**, 미국(팁·다이어터리 고지).
- 할랄 국가는 **할랄 인증 식당만 후보**로 필터 + 🕌 뱃지 + "무슬림 손님 배려" 문구.
- ⚠️ 식문화 데이터는 일반화된 참고용 → 실서비스는 큐레이션/현지 검수 필요.

### 16-5. 🍽 캐치테이블 실제 예약 — k-skill 연동 (실험적)
- [NomaDamas/k-skill](https://github.com/NomaDamas/k-skill)의 `catchtable-sniper` = 로그인된 크롬 세션으로 **자동 예약 시도**(공식 API 아님, 브라우저 자동화). MIT 라이선스, `/k-skill:catchtable-sniper` 호출.
- 판정: **기술적으로 붙일 수 있으나** ①캐치테이블 약관 위반 소지 ②사이트 변경 시 취약 ③사용자별 로그인 필요.
- 결론: **정식 기능 부적합 → "실험적 부가기능"으로만.** 정식 예약 동선은 여전히 **딥링크**(각 앱으로 연결).
