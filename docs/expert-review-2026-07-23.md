# 모심(moim) 6개 관점 전문가 점검 종합 보고

> 2026-07-23 · 디자인 / QA / 데이터 / 기획 / PM / 개발 6개 관점 병렬 점검 결과 종합

## 총평

이틀 만에 v0를 완성한 것 치고 완성도가 높다는 데 전원 동의. 브랜드 색 토큰화, 결정적 seed 설계, 파생값 방어 처리, 개인정보 원본의 저장소 제외 등 기본기가 좋다. 다만 여러 전문가가 독립적으로 같은 지점을 지목한 교차 이슈가 있고, 그중 2건은 서비스의 존재 이유와 직결된다.

## 🔴 교차 검증된 핵심 이슈 (여러 전문가가 동시 지목)

### 1. 가중치 배정 로직이 통째로 데드코드 (데이터 + QA 공통, 심각)

`assign.ts:15-23`과 `parse-xlsx.mjs:69-77`의 `LUNCH_WEIGHT`/`DINNER_WEIGHT` 키는 `mugyodong-bugeoguk` 같은 슬러그인데, 실제 `restaurants.json`의 id는 `R001`~`R166` 형식이라 **20개 키 전부 매칭 실패** → `weights[r.id] ?? 1`이 항상 기본값 1로 폴백. "인기 식당이 자연스럽게 편중되도록"이라는 설계 의도가 전혀 작동하지 않고, 방문횟수 분포는 순전히 난수의 우연. 현재 커밋된 `visits.json`은 구버전 파이프라인 산출물(1,139건)이라 정상처럼 보일 뿐, 엑셀 업로드나 `npm run data` 재실행 순간 평평한 분포가 드러난다.

### 2. "실적 기반 추천"이라는 핵심 가치의 신뢰 문제 (기획 + PM 공통, 심각)

엑셀에 식당명이 없어 거래를 더미 식당에 임의 배정하므로, 화면의 "방문 9회"는 알고리즘이 만든 숫자다. 임직원이 "나는 저기 안 갔는데 왜 1위지?"라고 느끼는 순간 전체 신뢰가 무너진다. **가맹점명 컬럼 확보가 서비스 성립의 전제 조건**이며, 불가하면 "방문횟수"를 "임직원 인기도(추정)"로 리프레이밍하고 근거를 명시해야 한다.

### 3. 개인정보 리스크 (데이터 + PM 공통, 심각)

- **마스킹 로직 결함**: 2글자 이름(`김구`)은 첫·끝 글자만 남기는 규칙(`parse-xlsx.mjs:40-43`, `xlsx.ts:15-19`)상 **원문 그대로 노출**. 한국에서 가장 흔한 2음절 성명이 마스킹을 우회하는 구조.
- `visits.json`에 카드사·계정과목·금액·날짜가 묶인 실데이터 112건이 저장소·배포에 포함. 소규모 조직에서는 부분 마스킹으로도 재식별 가능성(카드사+금액+날짜 조합)이 있다.

### 4. `setup.ps1` 원클릭 설치의 보안 표면 (PM, 심각)

`irm | iex` 원격 실행 + localhost:43110 브리지 + 시작프로그램 자동 등록 + `Access-Control-Allow-Origin: *`(+ Private Network Access) 조합은 임의 웹페이지가 로컬 PC에서 프로세스를 띄울 수 있는 표면을 만든다. 사내 PC 대상으로 보안팀 승인이 사실상 불가능. 발표에서는 "개념 시연"으로만 라벨링하고 실제 설치는 배제할 것. 실서비스 전환 시 localhost 브리지 트랙 전체를 분리하고 공식 예약 API 파트너십으로 대체 검토.

### 5. 가짜 "예약 완료" UX (기획 + 디자인 공통)

`ReservationForm.tsx:18-21`의 `setTimeout` 1.5초 후 무조건 성공 표시는 실사용 시 강한 배신감을 준다. 네이버/캐치테이블/스나이퍼 3중 예약 경로가 한 화면에 병렬 노출되어 "실제로 무엇을 눌러야 예약이 되는가"의 의사결정 마찰이 크다.

## 🎨 디자인 전문가

### 강점

- `globals.css:4-15` `@theme`로 SK E&S CI를 Tailwind 토큰으로 중앙화, 다크모드 이중 비활성화로 색 틀어짐 예방
- 방문횟수를 카드 우측 `text-lg font-black`으로 최우선 배치(`RestaurantCard.tsx:59-62`), TOP3 rank 배지 — 제품 컨셉과 정보 위계 일치
- 지도 핀 크기/숫자를 방문횟수에 비례, 지도 타일만 grayscale 처리해 컬러 핀 분리, 범례 완비(`KakaoMap.tsx:102-188`)
- 활성 상세필터 개수 배지("상세 필터 (3)"), 새로오픈/김영란법 판정 기준 인라인 안내
- 결과 0건 빈 상태 처리 존재(`MoimApp.tsx:579-583`)

### 문제점

**높음**
- 터치 타깃 44px 미달 다수: 시트 닫기 ~28px(`DetailSheet.tsx:48-53`), 정렬 버튼 ~26px(`MoimApp.tsx:529-538`), 뷰 토글 ~28px, 빠른 토글 칩 ~30px
- aria-label/role/aria-pressed/aria-expanded 사용 **프로젝트 전체 0건**. 지도 핀은 클릭 div/SVG라 키보드 조작 불가
- 바텀시트: `role="dialog"`/aria-modal 없음, 포커스 트랩·ESC 닫기 없음, 드래그 핸들은 장식일 뿐 스와이프 디스미스 미구현, max-height/overflow 제한 없어 긴 콘텐츠 시 하단 액션 유실 가능

**중간**
- safe-area(노치/홈 인디케이터) 대응 전무 — viewport-fit=cover, env(safe-area-inset-*) 없음
- `text-slate-400` 보조 텍스트가 아이보리 배경 위 11px~xs 크기로 광범위 사용 → WCAG AA(4.5:1) 미달 가능성
- `#fffdf8` 20회+, `#3d0b12` 다수가 토큰 없이 인라인 반복 — 값 변경 시 전수 수정 필요
- sticky 헤더에 필터가 누적되어 세로로 매우 길어짐, 결과 수 피드백이 헤더 밖에 있어 필터→결과 인과 연결 약함

**낮음**
- 예약 불가 식당의 "예약 문의" 버튼이 비활성처럼 보이나 클릭됨(어포던스 불일치, `DetailSheet.tsx:112-121`)
- NewPlaceCard와 RestaurantCard 시각 규격 불일치로 스캔 리듬 끊김
- KakaoMap/MapView 범례 규칙 불일치, `text-[10px]`/`text-[11px]` 초소형 텍스트 광범위

### 개선 TOP 5

1. 터치 타깃 `min-h-11`(44px) 일괄 적용 + 시트 드래그 디스미스/ESC (난이도 하~중)
2. 시트에 `role="dialog"` + 포커스 트랩 + `max-h-[85svh] overflow-y-auto` + safe-area pb (중)
3. `#fffdf8`/`#3d0b12`를 `@theme` 토큰으로 승격 후 전수 치환 (중)
4. aria-pressed/aria-expanded + 지도 핀 키보드 접근 (중)
5. 상세필터를 별도 바텀시트로 분리해 헤더 밀도 축소 + 실시간 결과 피드백 (상)

## 📋 기획 전문가

### 강점

- 방문횟수 1순위 정렬 + "최근 법인카드 사용 내역"(날짜/계정/금액)은 일반 맛집 앱이 가질 수 없는 사회적 증거 — 서비스의 진짜 무기
- 목적→회계 계정 자동 매핑(점심=의욕관리비/회식=경상회의비/접대=접대비) + 김영란법 근거 배너 — 사내 재무 규정을 소비자 UX로 번역한 성숙한 기획
- 데이터 부재에 정직: synth 플래그, "시연용 예시 데이터" 배지, features 미기입 시 boost 0점 처리

### 문제점

- **치명적**: 실적 기반 프레이밍의 신뢰 붕괴 위험 (교차 이슈 2)
- **높음**: `genderBoost`("여성→일식/양식/동남아", "남성→한식/중식")·`ageBoost` 하드코딩(`MoimApp.tsx:71-75`)은 실적이 아닌 고정관념의 코드화. 사내 서비스에서 성별 음식 추천은 정치적 리스크까지 있는 전형적 과잉 기능
- **높음**: 접대 시나리오 미완결 — features가 실데이터에 undefined라 부스트가 실질 무효, 예약 확정→참석자 공유→캘린더 등록이 전부 부재
- **중간**: 가짜 예약 완료 (교차 이슈 5) + SniperLauncher는 PC 설치·PowerShell·크롬 확장·로그인을 요구해 99%가 못 넘는 문턱 — 데모 화제성 대비 실사용 가치 과소
- **중간**: 매일 재방문할 후크 전무 — 단방향 조회 도구, "내 이력" 기반 개인화 없음
- **낮음**: 회식/접대는 팀 행위인데 앱은 철저히 개인 뷰 — 팀 저장/공유/투표 개념 부재

### 제안 TOP 5

1. 가맹점명 확보 → 실적 진실성 복구, 불가 시 "인기도(추정)" 리프레이밍 (가치 최상/공수 상)
2. 연령/성별 부스트 제거 → "내 방문 이력 기반 개인화"로 대체 (상/중)
3. 예약 UX 정직화 + 예약 정보 팀 공유·캘린더 등록 링크 (상/중)
4. "오늘의 점심" 홈 — 요일/최근 미방문 기반 3~5곳 로테이션 추천 (상/중)

- fflate + 정규식 경량 엑셀 파서로 번들 절약 + 파싱 즉시 마스킹

### 문제점

**높음**
- **JSON 약 250KB(restaurants 108KB + visits 134KB + newPlaces 8KB)가 클라이언트 번들에 통째로 포함** — `'use client'` MoimApp이 lib/data를 import. 모바일 타깃에서 초기 페이로드·파싱 비용·TTI 치명적
- `RESTAURANTS` 상수가 모듈 최상위에서 동기 계산(`data.ts:131-133`) — 서버 컴포넌트에서 미리 계산했다면 0이 될 비용

**중간**
- `MoimApp.tsx` 626줄 거대 컴포넌트 — useState 20개, 필터/정렬/부스트 로직 + UI + 하위 컴포넌트가 한 파일. 상태 하나 바뀔 때마다 전체 리렌더
- 필터 상태 20개 개별 useState — 필터 추가 시 상태 선언·useMemo 의존성·activeDetailCount 3곳 동시 수정 필요
- `React.memo` 미적용 — 필터 토글마다 모든 카드 재생성, 카드 인라인 애니메이션 재실행 가능
- localStorage·에이전트 응답을 검증 없이 `as` 캐스팅 — strict 타입 안전성이 경계에서 무력화

**낮음**
- 매직 넘버 산재: `distM / 67` 3곳 중복, 합성 건수 130/70이 스크립트/런타임 미공유

- `KakaoMap.tsx` eslint-disable exhaustive-deps 2곳 — renderPins stale 클로저 잠재 위험
